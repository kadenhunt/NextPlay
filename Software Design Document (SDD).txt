1. Purpose and Scope[b]
1.1 Purpose
NextPlay is a web-based fantasy college sports platform that lets users create accounts, join or create leagues, draft players, set lineups, and compete in weekly matchups using real collegiate performance data. This Software Design Document explains how the SRS requirements are realized through the system’s UI flows, frontend/backend architecture, data model, and API interface contracts. It defines the major screens and navigation, the key runtime behaviors, and the modules responsible for implementing those behaviors. It also documents security, testing strategy, and deployment considerations needed to build and validate an MVP within the semester timeline.
1.2 Scope
In scope
* User authentication (register/login/logout) and session persistence
* League creation, join via invite code/link, and league membership management
* League lifecycle states (Created → Draft Scheduled → Draft In Progress → Season Active → Playoffs → Complete) with UI gating
* Snake draft with timer + auto-draft on timeout; draft recap/results
* Team roster + lineup management with lineup lock enforcement
* Player browsing/search/filtering and availability display
* Weekly matchups and standings display (read-only MVP scoring updates allowed via refresh/polling)
* League chat (basic messages, no moderation tooling required for MVP)
Out of scope
* True real-time scoring + draft synchronization via WebSockets (polling acceptable for MVP)
* Advanced trade system (multi-step offers, veto windows, counters) beyond a basic proposal flow
* Predictive insights / analytics features
* Cross-league global social features, notifications, and push alerts
* Admin console (suspend users, audit tooling) beyond basic logging
1.3 Definitions and Acronyms
Term
	Definition
	League State
	The lifecycle phase of a league that controls which features are available (Created, Draft Scheduled, Draft In Progress, Season Active, Playoffs, Complete).
	Commissioner
	The league owner/administrator responsible for configuring league rules, managing members, and controlling draft-related actions (start/pause, settings changes) as permitted by league state.
	Snake Draft
	 A draft format where the pick order reverses each round (e.g., 1→N then N→1).
	Auto Draft
	 A system-driven pick made when a user fails to pick before the timer expires, using rankings/availability rules.
	Lineup Lock
	The cutoff time after which lineup changes are restricted for a matchup/week (typically when games begin).

2. Constraints, Assumptions, and Design Goals
2.1 Constraints
* Platform constraints:
   * Web-only application
   * Supported browsers: latest 2 versions of Chrome, Edge, Firefox, and Safari.
* Course constraints:
   * 3 Hours a week in class; scope must fit a semester build.
   * Team size limits parallel work; we prioritize an MVP path: one sport end-to-end before expanding features.
   * Limited budget: project must run on free/low-cost tiers where possible (hosting, database, etc.).
* Data constraints: 
   * Player and stats data depend on third-party NCAA data sources (availability, rate limits, schema changes).
   * Data may not update in real time; scoring can be periodic refresh for MVP.
   * Data completeness varies by sport/season (missing injury/status fields, inconsistent IDs across sources).
* Security constraints:
   * Authentication must use secure practices (hashed passwords, HTTPS, server-side authorization checks).
   * Store minimal PII (email + display name). No sensitive student data required.
When constraints conflict with features, priority is: correctness → security → usability → performance → extra features.
2.2 Assumptions
* Users have reliable internet access while drafting and setting lineups.
* External sports data sources will remain available for the semester and provide stable identifiers for players/teams most of the time.
* The MVP can treat stats updates as “near real-time” via polling/refresh rather than true live streaming.
* League members act in good faith (we don’t need anti-cheat or moderation systems for MVP).
* Time is stored in a consistent timezone strategy (UTC in backend; local display in UI).
* Invite links/codes can be assumed shareable and not “high security” beyond random token + expiry
2.3 Design Goals
* Maintainability: 
   * Keep a modular structure (auth, league, draft, roster, matchups) with clear API boundaries so features can be expanded without rewriting core logic.
* Usability: 
   * Optimize core flows (create/join → draft → set lineup → view matchup) with predictable navigation, clear empty states, and minimal clicks for common actions.
* Correctness:
   * Enforce server-authoritative rules for draft turns, roster limits, and lineup locks; database constraints prevent duplicate ownership and inconsistent states.
* Security: 
   * Use secure authentication, role-based authorization (commissioner vs member), and protect endpoints from unauthorized access; minimize stored personal data.
* Performance: 
   * Pages should feel fast under normal load; use pagination/filtering for player lists and caching for external data to avoid rate-limit issues.

3. System Context and Overview[c]
The system context view defines the operational boundaries of the NextPlay Fantasy College Sports Framework and illustrates how the system interacts with external actors and supporting systems. This section will provide an advanced representation of how the NextPlay Fantasy College Sport framework operates within its environment, clarifying integration points with external systems and identifying its primary stakeholders. Additionally, the context view does not describe core implementation details; instead, it demonstrates how the system communicates with users and support services. 


3.1 System Summary
The app is a web-based fantasy college sports platform that enables users to manage, create, and participate in fantasy leagues based on college athletic competitions such as basketball and football. The app allows users to create an account, join or create a league, manage team rosters, compete in weekly matchups, and participate in drafts. Moreover, the app retrieves official player statistics from external sports data providers and applies a scoring engine to calculate fantasy points and league rankings. The initial users of the system include fantasy league participants, league commissioners, system administrators, and guests. 
3.2 Stakeholders and User Roles
   * User capabilities include:
   * Register and authenticate an account.
   * Join or accept invitations to leagues.
   * Participate in live or scheduled drafts.
   * Select and manage team rosters.
   * Set weekly starting lineups.
   * Propose, accept, or reject trades.
   * View player statistics and scoring updates.
   * Communicate within league chat (if enabled).
   * Track standings, matchups, and performance history.


   * Commissioner capabilities include:
   * Create and configure leagues.
   * Define league rules (scoring format, roster size, draft type).
   * Schedule draft sessions.
   * Approve or veto trades (if league rules allow).
   * Remove inactive users from leagues.
   * Manage league visibility (public/private).
   * Reset or modify league settings before the season starts.

      * Guest: Capabilities include:
      * Browse public league information (limited access)
      * View general player statistics
      * Register for a full account
      * Explore platform features before joining



         * Administrator: 
The administrator role is responsible for overall system oversight and operational management.
Capabilities include:
         * Manage user accounts (suspend, activate, reset credentials)

         * Monitor system logs and audit activity

         * Manage integration settings with external sports data providers

         * Maintain database integrity and perform backups

         * Configure global system settings

         * Enforce security and compliance policies

Administrators operate within the highest privilege level but remain subject to authentication and auditing controls.

Context notes
            * External Systems
            * Sports Data Source:
The platform integrates with external sports data providers such as ESPN or Sportradar to retrieve official collegiate player statistics, game results, and performance metrics. These external services supply authoritative data used by the scoring engine to compute fantasy points and update weekly matchups. The system does not generate official sports statistics; it consumes and processes validated external data feeds.
            * Authentication Provider (Optional):
            * If third-party authentication is supported, identity services such as Google OAuth may be utilized to securely verify user credentials. If external authentication is not implemented, secure credential validation and token management are handled internally within the system boundary.



Trust boundary:
Trust boundaries represent security enforcement zones where authentication, authorization, and validation controls are applied.
               1. User-to-Application Boundary:All user interactions occur through secure HTTPS connections. Authentication is enforced at this boundary through credential verification and token validation (e.g., JWT). Unauthorized or unauthenticated requests are rejected before accessing protected resources.

               2. Application-to-Database Boundary: The backend enforces role-based access control before executing database operations. Authorization logic ensures users can only access or modify data associated with their account or league. Direct database access from external actors is strictly prohibited.

               3. Application-to-External API Boundary :Communication with external sports data providers occurs using secure API keys and encrypted channels. Responses are validated and sanitized before being processed by internal scoring services to prevent data integrity risks.

These trust boundaries ensure confidentiality, integrity, and controlled access across all system interactions while maintaining a clear separation between trusted internal components and external entities.

4. Architecture Overview


4.1 Architecture Summary
NextPlay follows a client–server architecture with a modular backend design. The frontend handles user interaction and communicates securely with the backend API. The backend processes business logic, authentication, league management, drafting, and scoring operations. A relational database stores all persistent data, including users, leagues, teams, and statistics. The system integrates with external sports data providers to retrieve official collegiate player statistics. This architecture supports scalability, security, and maintainability while keeping deployment simple
4.2 High-Level Containers
Figure ARC-01: Container Diagram
  
Containers
                  * Frontend:
Handles user interface, user interactions, and secure communication with the backend.
                  * Backend API:
Implements business logic, enforces authentication and authorization, manages leagues and scoring, and communicates with the database and external APIs.
                  * Database:
Stores persistent data such as users, leagues, rosters, draft history, and scoring results.
                  * External Data Provider:
Supplies official collegiate sports statistics used for fantasy scoring calculations.
4.3 Key Architectural Decisions
Decision
	Choice
	Rationale
	API style
	REST (JSON over HTTP)
	Simple to implement, easy to test with Postman, maps cleanly to screens/modules, and works well for a web MVP.
	Data store
	PostgreSQL (relational)
	 Supports the ERD/entities (users/leagues/teams/picks/matchups) and supports joins/constraints for correctness.
	Real-time updates
	Polling for MVP; WebSocket/SSE later
	Polling is simpler + reliable under time constraints; draft/chat can be upgraded to real-time without changing UI structure.
	Auth
	JWT (stored in HttpOnly cookie) + server-side authorization
	JWT works with stateless APIs; HttpOnly cookie reduces token exposure; backend remains the authority for permissions.

5.5 UI Component Conventions

 Figure UI-COMP-01: Component Kit Overview
   
  

                           * Buttons:
                           * Primary: Used for the main action on a screen (Create League, Join, Draft Player, Save Lineup). Only one primary per screen when possible.
                           * Secondary: Used for supporting actions (Cancel, Back,View Details).
                           * Disabled: Used when requirements are not met (invalid form, locked league state, not user’s draft turn). Disabled buttons include a tooltip or inline hint when applicable.
                           * Destructive: Used for irreversible actions (Remove Member, Reset Invite Code, Drop Player), requires confirmation.


                              * Inputs:
                              * Default: Labels always visible; helper text shown for complex fields (draft timer, team count).
                              * Error: Inline validation message directly under the field; submission blocked until fixed.
                              * Inputs support “loading” state where applicable (e.g., validating invite code).

                                 * Status indicators: 
                                 * League lifecycle is displayed with a consistent state pill: Created, Draft Scheduled, Draft In Progress, Season Active, Playoffs, Complete.
                                 * State pills are shown on League Home and Draft-related screens, and used to explain gating.
                                 * Alerts:
                                 * Info: Used for guidance (e.g., “Draft starts at 7:00 PM”).
                                 * Warning: Used for lock/state restrictions (e.g., “Lineup locked for Week 3”).
                                 * Error: Used for failures (network errors, permission denied).
                                 * Alerts appear as inline banners on the page; quick confirmations like “Save” may use toast notifications.
                                 * Lists and tables:
                                 * Lists and tables follow consistent row structure and spacing.
                                 * Roster rows show player name, position, real team, status, and actions.
                                 * Standings rows show team name, W–L, points for/against, and rank.
                                 * Tables support empty states (“No matchups yet”) and loading placeholders.

	6.3 Key Runtime Scenarios
                                    * Scenario R-01: [Draft pick submitted]
                                    * Trigger: User clicks “Draft” on a player in UI-13 Draft Room.
                                    * Preconditions: League state = Draft In Progress; user is a league member; it is the user’s turn; player is available (not already drafted).
Steps:
                                    1. Frontend submits pick request with leagueId, playerId.
                                    2. Backend validates token and membership.
                                    3. Backend checks current pick owner == user AND timer not expired.
                                    4. Backend checks player not already drafted and roster limits allow selection.
                                    5. Backend creates DraftPick record and assigns player to the user’s team roster.
                                    6. Backend advances the draft state to next pick and returns updated draft snapshot.
                                    7. Frontend updates draft board, roster preview, and turn indicator. Postconditions: Pick is persisted; player marked unavailable; current pick advances.
Failure cases:
                                    * Not user’s turn → 403/409 with message (“Not your pick”).
                                    * Player already drafted → 409.
                                    * Draft paused/ended → 409.
                                    * Network failure → UI shows retry + local “pending” cleared.
                                    * Scenario R-02: [Auto-pick on timer expiration]
                                    * Trigger: Pick timer expires for current pick owner
                                    * Preconditions: League state = Draft In Progress; timer reaches 0; backend has draft order and pick index.
Steps:
                                       1. Backend detects timer expiration (scheduled job / server timer / draft tick endpoint).
                                       2. Backend selects best available player using priority rules:
                                       * If user has a queue → highest-ranked available from queue.
                                       * Else → highest-ranked available player overall (by default ranking).
                                       3. Backend creates DraftPick and assigns player to team roster.
                                       4. Backend advances to next pick and resets timer.
                                       5. Frontend (polling or push) receives updated draft state and shows “Auto-pick occurred” banner.
Postconditions: A valid pick is made and persisted; draft continues without stalling.
Failure cases:

                                          * No players available → draft ends early with “Draft Complete (insufficient pool)”.
                                          * External ranking unavailable → fallback to internal default ordering.
                                          * Scenario R-03: [Lineup lock enforcement]
                                          * Trigger: User attempts to save lineup in UI-15 My Team OR lock time passes.
                                          * Preconditions: User is team owner; league state = Season Active; week is known; lock timestamp exists.
Steps:
                                          1. Frontend requests GET /leagues/:id/rules (or lock info) and displays lock countdown.
                                          2. User edits starters/bench and clicks Save.
                                          3. Backend validates lock: if current time >= lock time, reject update.
                                          4. If before lock, backend validates lineup constraints (positions/roster membership).
                                          5. Backend persists lineup and returns updated lineup object.
Postconditions: Lineup saved if before lock; after lock, lineup remains unchanged and read-only.
Failure cases:
                                             * After lock → 409 “Lineup locked for Week X”.

                                             * Invalid position constraints → 400 with field-level message.

                                             * Player not on roster → 400/404.


                                                * Scenario R-04: [Add or drop player with roster full]
                                                * Trigger: User clicks Add/Drop from UI-18 Players or UI-15 My Team.
                                                * Preconditions: League state allows transactions (Season Active); user is team owner; roster size limit defined.
Steps:
                                                1. Frontend initiates add/drop request.
                                                2. Backend checks roster size:
                                                * If adding and roster is full → require drop+add as one atomic transaction.
                                                3. Backend validates player availability (not already rostered elsewhere if your rules require exclusivity).
                                                4. Backend updates roster and logs Transaction record.
                                                5. Backend returns updated roster.
Postconditions: Roster updated correctly; transaction recorded.
Failure cases:
Roster full and no drop selected → 409 “Roster full”.
                                                   * Player unavailable → 409.
                                                   * After transaction deadline/lock → 409.

8. API and Interface Contracts
8.1 API Style
                                                                                                                     * Protocol: We will be using HTTP to encode and deliver our requests throughout the program, this will be used to fetch live scores from ongoing games as well as fetching player stats and information to be displayed for the user.

                                                                                                                     * Format: We will be using a JSON format method to be delivered with our HTTP requests to make a simple and easy to encode and then decode our responses.

                                                                                                                     * Auth: Once each user logins and receives a positive authentication, we will generate a JSON Web Token to store that authentication so the user maintains authentication during each session. This token will be stored using the browser’s cookie storage to help protect the token and ensure consistent access.

                                                                                                                     * Error format: We will generate error messages into a JSON Map to deliver to the program’s front in order to be able to decode and display the user’s error and inform them on what feature may have broken or what problem may have occurred that created a breakpoint within the software.

9. Security and Access Control[g][h]
Access control is a core design concern and should be addressed explicitly.
9.1 Authentication
Method: JWT-based authentication. Users authenticate using email and password. Upon successful login, a signed JWT access token is issued.
Session Management Approach: Stateless session management using short-lived access tokens and refresh tokens. Tokens include expiration timestamps and are validated on each request.
________________


9.2 Authorization
Roles:
                                                                                                                           * Member: Can join leagues, draft players, manage roster, create trades, and participate in chat.

                                                                                                                           * Commissioner: All member privileges plus league creation, draft control, trade approval (if enabled), and league settings management.

Enforcement Points:
 Authorization is enforced through backend middleware for token validation and role verification, with additional service-layer checks to restrict access to league-specific resources.
________________


9.3 Data Protection
Password Storage: Passwords are hashed using BCrypt with salting to prevent brute-force and rainbow table attacks.
Transport Security: All client-server communication is encrypted using TLS (HTTPS).
Sensitive Fields: Passwords, email addresses, authentication tokens, trade transactions, private league data, and chat messages.

                                                                                                                                       * Index foreign keys (league_id, user_id, player_id).
                                                                                                                                       * Index frequently queried fields (league_id in Matchups, sport_id in Player).
                                                                                                                                       * Unique constraints on draft picks to prevent duplicate ownership.\


                                                                                                                                          * [Horizontal scaling approach if relevant]

11.3 Logging and Monitoring
                                                                                                                                                      * Logging approach:
For operational monitoring, auditing, and troubleshooting, NextPlay shall use server-side logging to capture important system events and errors. Only authorized administrators shall be able to access the safely stored logs. User authentication activities, league setup and configuration modifications, draft actions, roster transactions, score updates, and external API failures are all considered logged events.


                                                                                                                                                      * Basic metrics: 
The system shall track basic operational metrics, including:
                                                                                                                                                         * API request latency: average response time under 300 ms under normal load and < 1 second during peak usage.

                                                                                                                                                         * Error rate: maintain < 2% failed requests (HTTP 4xx/5xx combined) under normal operation; log all 5xx errors.

                                                                                                                                                         * External API performance: record success/failure rate and response time for each external sports API request.

                                                                                                                                                         * Database query latency: database queries shall complete within 200 ms under normal load and 500 ms during peak usage.

                                                                                                                                                         * Draft pick processing time: draft pick submissions shall be processed and confirmed within 2 seconds of user selection.

                                                                                                                                                         * Scoring update frequency: the system shall update scoring at least every 5 minutes during active games and finalize matchup results within 10 minutes after official game completion data is available.
