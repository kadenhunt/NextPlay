1. Introduction
   1. Purpose 
This product will be marketed towards current sports fans and especially NCAA fans as well as Fantasy sports players to try and offer a new experience with teams that haven’t been featured in the fantasy sports world. This document can be used as a guide for the software development team to ensure they stay on track with the development process and are meeting each requirement outline. As well, this document can be used to give market staff an effective outline for the various included features and design of the app for advertising. It will also give the testers a baseline for each of the different feature’s intended functionality to help guide their testing process. 
   2. Project Scope
For our project, we will design a framework to allow users to play fantasy sports with college sports teams, including football, basketball, and baseball . This app will allow users to schedule and perform a draft on their chosen team before the official start of the season. Users will be able to control and establish their roster and lineup based on how they want to build their team. Then throughout the season, the application will report the results from each player and their performance in order to calculate each user’s point totals to determine the winner of that week. As well, users will be able to create trades and chat with other players in their league. There will be rankings reported each week as well as a final playoff for the top members of the league to decide the overall winner.
   3. References
We will be utilizing multiple API services in order to accurately and effectively source our data for each of the sports we will be featuring on our application. For NCAA Football, we will be using the API provided by https://api.collegefootballdata.com/#/games/GetScoreboard to fetch our team rosters and player stats for college football. For our game results for basketball and football, we will be using https://github.com/henrygd/ncaa-api to be able to calculate results for users each week.
2. Overall Description
<This section presents a high-level overview of the product and the environment in which it will be used, the anticipated users, and known constraints, assumptions, and dependencies.>
   4. Product Perspective
NextPlay is intended to give users an interactive fantasy sports experience based on professional college sports including football, baseball, and basketball. The software is designed to be used independently, enabling users to pick teams before the season begins, manage fantasy leagues, and make smart roster choices all season long. The NextPlay app is a brand-new platform with an emphasis on usability, competition, and engagement rather than a replacement for an already-existing application. In order to track player performance and compute fantasy scores, the application works with a larger ecosystem that includes other data sources for college sports statistics. A user interface for league administration, drafting, trade, and in-app communication is one of the main components of the system, together with backend services that handle rankings, scoring, and playoff logic. The system is built to facilitate future expansion to more sports or increased functionality, and it may interface with internal data services or third-party APIs to retrieve and process player statistics.
   5. User Classes and Characteristics
This product accommodates several user classes with varying levels of access and responsibility. The Fantasy League Participant is the most popular user class, accounting for the vast bulk of system activity.


* Fantasy League Participants (Primary Users) are end users who join leagues to choose players, manage rosters, arrange lineups, monitor scores, and engage with others. They are familiar with fantasy sports and want an easy-to-use interface with accurate, fast information.
* League Commissioners: Users who create and manage leagues. In addition to participant features, they configure league rules, manage memberships, and oversee league operations.
* System Administrators:
 Technical users responsible for maintaining system availability, security, data integrity, and integrations with external data sources. They do not participate in leagues.


   6. Operating Environment
The system will be available as a web-based application on modern desktop and mobile devices with an internet connection. Client platforms supported include desktops, tablets, and smartphones that run current versions of major operating systems such as Windows, macOS, Android, and iOS, as well as modern web browsers.


The program will be hosted on cloud-based servers in the United States, ensuring availability, scalability, and performance. Application servers and databases will use industry-standard server operating systems and managed database platforms. External data sources can be accessed via secure APIs to acquire college sports statistics.


The system is intended to work alongside common third-party services including authentication providers, notification services, and data analytics tools. In addition to typical consumer devices, end users do not require any specialized gear.
   7. Design and Implementation Constraints
The system must be developed as a web-based application compatible with modern browsers and standard consumer devices, which limits the use of platform-specific or proprietary client-side technologies. The application design is constrained by the availability, reliability, and usage limits of external data sources used to retrieve college sports statistics.
The system must comply with applicable data privacy and security standards, influencing authentication, data storage, and communication mechanisms. Performance constraints require the system to handle concurrent users efficiently during peak activity periods, such as drafts and game days.
The implementation is further constrained by the requirement to use widely supported programming languages, frameworks, and database technologies to ensure maintainability, scalability, and long-term support. Integration with third-party services must follow their defined APIs and usage policies.
   8. Assumptions and Dependencies
Assumptions:
The following assumptions will be made during the development of this project and may impact the stated requirements if they change:
   * It is assumed that reliable and accurate college sports statistical data and information will be available through external data sources such as public or third-party APIs, throughout the sports seasons.
   * In terms of usage, users are assumed to have a reliable internet connection and to use a modern web browser or supported mobile devices capable of running the application.
   * The application assumes that college sports schedules, scoring rules, and player rosters stay relatively stable during a season with only minor changes that can be handled through configuration updates.
   * It is assumed that users have essential familiarity with fantasy sports concepts, including scoring, rosters, draft, and weekly matchups.
   * The application assumes enough server resources are available to support expected user traffic, especially during peak usage time, such as game day and draft.
   * It also assumed that the authentication and user account services (e.g., login services or email verification) function reliably and securely.
Dependencies: 
The project depends on the following external factors, which are outside the direct control of the system:
   * Accessibility and uptime of third-party data providers supplying college sport statistics, player performance data, and schedules. 
   * Stability and performance of hosting infrastructure, database platforms, or cloud services used to deploy and operate the application. 
   * Changes in external APIs, including data formats, rate limits, or access restrictions, which may require a system update. 
   * Compliance with applicable data privacy, security, and usage regulations regarding user information and third-party data sources. 
   * Timely updates to operating browsers, systems, and device platforms to ensure compatibility with the application.

   9. User Account Management and Authentication
   1. Description
Priority: High
This feature allows users to create and manage personal accounts within NextPlay, enabling secure access to the platform and persistent storage of user-related data. User accounts allow individuals to join and manage multiple fantasy leagues, access their teams from different devices, and retain league history across seasons. This feature supports personalized experiences and ensures that user data remains consistent across sessions.
   2. Stimulus/Response Sequences
When a user selects the option to register or log in, the system prompts the user to provide authentication credentials. Upon successful authentication, the system grants access to the user dashboard and displays leagues associated with the account. If authentication fails, the system notifies the user and requests corrected input.
   3. Functional Requirements
   * The system shall allow users to create accounts using unique credentials.
   * The system shall authenticate users before granting access to league and roster data.
   * The system shall allow users to participate in multiple leagues simultaneously.
   * The system shall maintain user session data across supported devices and browsers.
   1. Fantasy Sport Selection
   1. Description
Priority: High
This feature allows users to select the college sport they wish to participate in when creating or joining a fantasy league. Supported sports currently in scope are football, basketball, and baseball. The selected sport determines the available teams, players, scoring rules, and seasonal structure associated with the sport calendar. By supporting sport selection at the league level,  the system gains flexibility along with modular expansion and clear separation of league configurations across different sports.
   2. Stimulus/Response Sequences
When a user starts the league creation process, the system prompts the user to select a supported sport. Upon selection, the system will load the corresponding teams, players, schedules, and scoring configurations associated with that sport.
   3. Functional Requirements
   * The system shall allow users to select a supported college sport when creating a league.
   * The system shall associate each league with exactly one sport for its duration. The system shall load sport specific player pools, schedules, and scoring logic based on the selected sport.
   * The system shall prevent users from selecting unsupported or unavailable sports.
   1. League Creation and Secure Invites        Description
   1. Description
Priority: High
This feature allows users to create private fantasy leagues and invite other participants in a secure and controlled manner. League creation establishes the core configuration for gameplay, including sport selection, league size, and draft settings. Secure invites ensure that only invited users can join a league, making the process secure and preventing unauthorized league access.
   2. Stimulus/Response Sequences
When a user chooses to create a league, the system prompts the user to define basic league parameters and generates a unique league instance. The system then provides the user with invite options like invite links or codes to share with league participants. When an invited user accesses an invitation, the system requires the user to authenticate or create an account before validating the invite. Upon successful validation, the system adds the user to the league. If the invite is invalid, expired, or the league is full, the system notifies the user and denies entry.
   3. Functional Requirements
   * The system shall allow authenticated users to create fantasy leagues.
   * The system shall generate a unique identifier for each league.
   * The system shall support secure invitation mechanisms that restrict league access to invited users only.
   * The system shall require invited users to authenticate or create an account before joining a league.
   * The system shall validate invitations before allowing users to join a league and prevent access when invites are invalid, expired, or the league has reached its maximum number of participants.
   1. League Settings and Commissioner Controls
   1. Description
        
        Priority: High


A league commissioner can use this feature to control and modify league settings during the league's existence. League regulations, roster configurations, scoring schemes, and member involvement are all under commissioner control. These regulations guarantee consistent league operations and fair gameplay.


   2. Stimulus/Response Sequences
Configurable options, including roster size, scoring regulations, league privacy, and member management, are shown by the system when a league commissioner visits league settings. The commissioner makes adjustments and sends them. After verifying the updates, the system applies them if authorized. The system rejects the modification and shows an error notice if it is prohibited by the league state (for example, after the draft starts).
   3. Functional Requirements
   * The league creator should be recognized as commissioner by default.
   * The commissioner should be allowed to update/change league settings.
   * The system will only allow the commissioner to update/change the settings.
   * The system should validate the settings and reject invalid changes.
   * Users should be notified when settings are updated/changed.
   1. Draft System (Snake Draft / Auto Draft)
   1. Description


        Priority: High
Teams may select players through a pre-season draft process enabled by this feature. Leagues can choose between human player selection and automatic roster assignment based on predetermined rankings, thanks to the system's support for both snake draft and auto-draft modes.
   2. Stimulus/Response Sequences


When a draft begins, the system should display the draft interface, including picking orders, team rosters, and player availability. In a snake draft, teams select in turns/sequence, and the order of selection reverses in each round. If a user fails to select a player within the allotted time, the system will assign a random player to the team. Thus, in auto-draft mode, the system will assign the players automatically to each team.


   3. Functional Requirements


   * The system should allow the commissioner to choose draft type and settings.
   * The system should use a reversing pick order in each round.
   * The system should be able to support auto-draft mode by assigning players automatically to each team.
   * The system should be able to prevent assigning one player to multiple teams.
   1. Roster and Lineup Management
   1. Description


        Priority: High


This feature will allow users to manage their weekly team lineups and rosters for their fantasy team. The users will be allowed to add/remove, and organize players on their rosters and appoint active starters for the upcoming scoring period. This feature makes sure that points are calculated based on user selected lineup and league rules.


   2. Stimulus/Response Sequences


Team lineups and current rosters should be displayed when users access their team page. Players can be added/dropped, or moved between active and bench positions by the user. The system validates roster and lineup changes against league rules and deadlines. The roster and lineup are updated by the system if they are valid. If the action is invalid (for example, the lineup is locked or the roster limit has been reached), the system rejects it and shows an error message.


   3. Functional Requirements


   * Users should be able to view their team lineups and rosters.
   * Users should be able to add/drop players from their rosters.
   * The system should enforce constraints for lineup position and roster size.
   * Users should be prevented from adding a player to their lineup who is selected by another team in the league.
   * Users should not be able to edit the roster and lineups after the deadline for lineup lock.
   1. Available Player Pickup/Drop
   1. Description


        Priority: High


This feature will allow the user to add the available player to the roster and drop the existing one during the season. Player pick/drop actions enable roster adjustments based on strategy, injuries, or performance, with the consideration of fair competition and league rules. 
   2. Stimulus/Response Sequences
        
With this feature, when a user observes the list for the player, the system will only show the available players' list. Users can add/drop players, and the system should be able to validate this transaction against league rules, deadlines, and roster size. If the transaction is valid, the system will update the roster. If the transaction is invalid, the system will throw an error message based on the circumstances (for example, the roster is full, or the player is unavailable).


   3. Functional Requirements
   * The system should only display the available player for pickup.
   * The system should allow users to add players to the roster based on league rules.
   * Users should be able to drop players from the rosters.
   * The system should enforce roster size for pick and drop. 
   1. Scoring and Weekly Matchups
   1. Description


        Priority: High
This feature will calculate fantasy points for players based on their real-world performance and weekly matchup outcomes. In the meantime, it will update the score for player statistics, weekly matchups, and league standings.
   2. Stimulus/Response Sequences
When the real-world game data becomes accessible, the system will retrieve the data by applying the league scoring rules. The system will calculate the scoring points for each player and aggregate points for each team in the league based on their active lineup. Finally, the system will compare the scores and determine the winner. 
   3. Functional Requirements
   * The system must retrieve players' data from external sources (API).
   * Player fantasy points should be calculated in regard to league rules.
   * Only active lineup points should be calculated for weekly team matchups.
   * The system must generate weekly matchups based on the league schedule.
   *    1. Standings and League Playoffs
   1. Description
Priority: High
This feature shows league standings based on matchup outcomes and records team performance throughout the season. Teams are ranked by the system based on their overall fantasy points, wins, and losses. The system selects the league winner at the end of the regular season and creates playoff games based on league standings.
   2. Stimulus/Response Sequences
After weekly matchups are completed, the system will update the league standings and team records. Users will be able to see a table containing team ranking and statistics. At the end of the season the system shall recognize the qualifying teams and generate playoff matchups. The system will update playoff results and promote winning teams to determine the champion. 
   3. Functional Requirements
   * The system shall update the standing based on the matchup results.
   * The system shall rank teams based on losses, wins, and tiebreaker rules. The system shall apply the following tie-breaker rules:
   * Head-to-head record between tied teams during the regular season.

   * Total cumulative fantasy points scored by each team during the season.

   * Highest single-week fantasy point total achieved during the season.

   * Lowest cumulative fantasy points allowed (points scored by opponents).

   * Random selection by the system if all other tie-breakers are equal.
      * League standings shall be visible to all members of the league.
      * The system shall determine the champion at the end of the season.

      1. League-wide User Chat and Matchup-Specific Messageboards
      1. Description
Priority: Medium
With the help of this feature, players will be able to interact with other league members through matchup-specific messageboards and league-wide chat. The league's user involvement and engagement will be improved by this functionality. While the matchup-specific message board will only allow users to discuss with their weekly opponents, the league-wide chat will function as a general discussion forum for all users. 
      2. Stimulus/Response Sequences
The system will show previous messages when a user logs into the matchup messageboards or league-wide chat. The system will capture and show new messages that the user writes and submits in real time. The message rules (TBD), such as being empty or going over the allotted length, will be implemented. 
      3. Functional Requirements


      * The system shall make the league-wide chat accessible to all league members.
      * The system shall make the matchup-message board accessible to all teams and authorized members.
      * The system shall use chronological order to display the messages.
      * The system shall save the message with a timestamp and the user identity.
      1. Weekly Fantasy Matchup Resolution
      1. Description
Priority: High
After player statistics are evaluated, this function resolves team scores to determine the ultimate result of weekly fantasy contests. The system records matchup outcomes for standings and playoff qualification, applies tie-breaker criteria when needed, and computes final team totals based on active lineups.
      2. Stimulus/Response Sequences
The system should freeze the match for resolution and compute the teams' final scores in accordance with league scoring regulations once the scoring period has concluded and player statistics have been updated and approved. The system will update the team records and standings, compare team totals, and declare a victor or tie. 
      3. Functional Requirements


      * The system shall recognize the end of the scoring period and trigger matchup resolutions.
      * The system shall apply tie-breaker rules if the scores are equal. The system shall apply the following tie-breaker rules:
      * Highest scoring starting player on each team for that matchup week.

      * Highest cumulative bench player points for that matchup week.

      * Highest projected team score prior to the matchup (if projections are available).

      * Declare the matchup as a tie if all tie-breaker conditions remain equal.

         * The system shall compare weekly matchups' scores and determine a win, a loss, or a tie.
         *  The system shall store the matchup results history.

         4. User Interfaces[a]
NextPlay will provide a responsive web interface for desktop and mobile browsers. The interface will focus on clarity, ease of navigation, and quick access to league critical information such as rosters, matchups, standings, and player performance. The interface will be organized into logical sections to support intuitive navigation and efficient access to prominent features.


The user interface shall:

         1. Software Interfaces
NextPlay will interface with external software systems to obtain college sports data required for fantasy league functionality. These systems might include third-party data providers  that supply information such as teams, players, schedules, game results, statistical performance, and player status updates. The system will consume structured data formats provided by these services and translate them into suitable representations for league management, scoring, and matchup evaluation. 


NextPlay will also interface with internal software components responsible for user authentication, league configuration, data persistence, and real time interaction features such as messaging. The data exchanged will include user profiles, league settings, rosters, scoring results, and communication content. Data transformations will be used as needed to ensure compatibility between external data sources and internal models.
Nonfunctional considerations for software interfaces include maintaining reliable data syncing, handling incomplete or delayed data updates and proper access controls to protect the system.

NextPlay does not require any specialized hardware interfaces and is intended to operate on standard consumer devices capable of running modern web browsers.
         3. Communications Interfaces
NextPlay will communicate over standard web-based networking protocols to support client-server interactions and exchange data with external services. Specific communication mechanisms will be defined during system design.
         6. Quality Attributes
         1. Usability
The app should be intuitive, simple to learn, and efficient for both new and returning users. Core operations including drafting players, managing rosters, establishing lineups, and evaluating scores must be conducted with few steps and excellent visual feedback. The user interface must have consistent layouts, vocabulary, and navigation across all displays in order to improve memorability.


The app must give users with clear error notifications and information to help them prevent, recognize, and recover from failures. To accommodate a diverse range of users and devices, accessibility considerations such as readable text, responsive design, and keyboard and touch capabilities must be included. The application must adhere to conventional online usability and accessibility criteria to offer a user-friendly experience.
         2. Performance
During peak usage periods, such as drafts and live game updates, the system must support concurrent users without exhibiting any obvious performance reduction. Under typical operating settings, standard user operations such as page navigation, roster updates, and score viewing must be responded to within an appropriate time frame.


Player statistics, scoring computations, and league standings must be processed and updated on a timely basis to maintain accuracy and consistency throughout the season. The system must be designed to scale as the number of users and leagues grows, while ensuring consistent availability and responsiveness.
         3. Security


User accounts and any sensitive data, such as personal information and login passwords, must be encrypted using industry-standard protocols during data transmission and storage. All data exchanged between clients and servers must follow secure communication protocols such as HTTPS with TLS encryption. Sensitive information saved in the database must be secured at rest using strong encryption algorithms (such as AES-256). Passwords should never be saved in plain text; instead, they should be securely hashed and salted with advanced cryptographic hashing techniques. Backup data should also be encrypted and securely kept, avoiding illegal access.




Regular monitoring and automatic notifications must be implemented to detect unwanted access attempts, suspicious behavior, data breaches, and system anomalies. To identify and minimize potential threats, intrusion detection techniques and periodic security assessments (such as vulnerability scanning and penetration testing) must be implemented. The system must also enable secure session management, which includes automatic session timeouts after inactivity and secure logout capability to prevent session hijacking. Together, these procedures assure the system's confidentiality, integrity, and availability.


User accounts and any sensitive data, such as login passwords and personal information, must be encrypted using industry-standard protocols during data transmission and storage. All data exchanged between clients and the server must follow secure communication protocols such as HTTPS with TLS encryption. Personal Information saved in the database must be secured at rest using strong encryption algorithms. Backup data should also be encrypted and securely kept, avoiding illegal access.


         4. Safety
By putting safeguards in place to protect user data, user interactions, and system integrity, the system must be designed to lower the risk of damage, loss, or injury occurring from its use. Additionally, regular data backup and recovery procedures will lower the chance of corruption or data loss, and secure authentication, role-based access control, and encrypted data transmission will be utilized to prevent unwanted access to accounts and sensitive information. To lessen the possibility of service outages during periods of high demand, the system must have safeguards against excessive system load, such as rate restriction and monitoring. To prevent destructive acts like data manipulation, system misuse, and invalid transactions, all user inputs must be checked. Additionally, safeguards must be in place to guarantee correct league status, scoring, and roster management in order to maintain fairness and prevent disputes.

Availability: 


With the exception of planned maintenance periods, the system must be accessible to users at least 99 percent of the time during the active sports season. Downtime must be kept to a minimum during moments of high demand, such as drafts and live game updates, and planned maintenance must be announced beforehand.


Reliability: 


With automated validation checks to identify and fix data inconsistencies, the system must reliably generate accurate results for league outcomes, standings, and scoring. To avoid extended disruption in the case of a system failure, fundamental functionality must be restored in a reasonable amount of time.


Efficiency and Performance:


Under typical operating circumstances, the system must respond to user requests—such as roster updates and score computations—in a reasonable amount of time. System stability and data quality will take precedence over non-essential features in performance improvement.
         7. Internationalization and Localization Requirements
NextPlay shall be designed with basic internationalization considerations to support future expansion beyond its initial user base. The initial release of the system will target English speaking users, but the application will allow users to select a preferred display language when supported.


User-facing text, labels, and messages shall be structured in a manner that allows for separation from core system logic, allowing additional languages to be added in the future without having to make major architectural changes. Date and time information shall be presented consistently.
         8. Other Requirements
8.1        Legal and Compliance Requirements
NextPlay will handle user account information in accordance with applicable privacy and data protection requirements. Personal data and user credentials must be safely kept and shielded from unwanted access. Only publicly accessible sports data or data acquired from approved API providers in compliance with their usage guidelines may be used by the system.


Sensitive personal information cannot be stored or displayed by the system beyond what is required for user account functionality.
8.2         System Logging and Audit Requirements
Logs of significant system events must be kept by the system to facilitate security, monitoring, and troubleshooting.


The system should log:


         * Events of user login and logout.
         * Configuration and creation of leagues.
         * Roster transactions and draft selection.
         * Matchup resolutions
         * System errors and failures
8.3        System Startup and Availability Requirements
During system setup, the necessary services, database connections, and external API connections must be initialized.


The system will gracefully manage external API availability issues and, if feasible, retry data collecting.


With the exception of planned maintenance, the system must function continuously during the sports season.

 The process where teams select players to build their roster before the season begins.

 Points given to players based on their real-world performance in accordance to league scoring rules.

 The active players who are selected from a team’s roster whose performance adds to the team’s weekly score.

 A match between two fantasy teams during a scoring period.
Roster
 The list of players assigned to a fantasy team.
Snake Draft
 A draft format where the pick order reverses in each round.

 A ranked list of teams based on wins, losses, and total fantasy points.

Appendix B: Analysis Models
B.1 Entity-Relationship Model:


The NextPlay system includes the following primary entities:


User
 Represents registered users who join fantasy leagues.
League
  A fantasy competition created by users.
Team
 Represents a user’s fantasy team in a league.
Player
 Represents real-world college athletes available to choose from.
Roster
 Shows the relationship between teams and players.
Matchup
 Shows weekly competitions between fantasy teams.
Score
 Shows fantasy points calculated based on player performance.
Relationships:
         * A User can join multiple Leagues.

         * A League has multiple Teams.

         * A Team belongs to only one User and one League.

         * A Team contains multiple Players through the Roster.

         * A Matchup is between two Teams.

         * Players create Scores based on performance data.


B.2 Data Flow Overview


The system processes data through the following sequence:
            1. External sports APIs supply player statistics, schedules, and game results.

            2. The system gets and stores player and game data in the database.

            3. The system applies league scoring rules to calculate fantasy points.

            4. The system updates team scores and resolves matchups.

            5. The system updates league standings and displays results to users.



B.3 System State Overview
A League progresses in the following states:
               * Created

               * Draft Scheduled

               * Draft In Progress

               * Season Active

               * Playoffs

               * Season Complete

Transitions between states occur based on draft completion, scoring updates, and season progression.
